# Multi-Order Cubes (MOC)

Reinforcement learning environment for robotic manipulation built with **Isaac Lab** and **Isaac Sim**, designed to train a robot arm to execute conditional object manipulation commands.

The project was originally developed as an experiment while learning the NVIDIA robotics ecosystem and exploring ideas related to **physical AI workflows**.

---

# Disclaimer

This repository is primarily a **learning project**.

The environment is functional but not production-ready and the training pipeline was not fully stabilized.

The code is shared mainly to document the experimentation process and lessons learned while working with Isaac Lab.

---

# Project Overview

This repository implements a custom reinforcement learning environment where a robot must manipulate cubes across discrete slots on a table.

The agent receives an instruction of the form:

```
(from_slot, to_slot)
```

Example:

```
(3,1)
```

Meaning:

> Move the cube currently located in slot 3 to slot 1.

The environment was designed to eventually support sequences of commands generated by higher-level reasoning models such as **Cosmos Reason**, where an external system produces manipulation plans and the RL policy executes them.

Example conceptual interaction:

```
Planner -> (2,3)
Policy executes movement
Policy -> NEXT
Planner -> (4,1)
```

The full intended manipulation pipeline was:

```
Reach → Grasp → Lift → Transport → Place → NEXT
```

However, during the project only the **Reach stage** was actively trained and stabilized.

# Environment Description

## Workspace

The environment consists of a table with **four fixed cube slots**.

```
slot 1
slot 2
slot 3
slot 4
```

Only **three cubes are present per episode**, leaving one slot empty.

Cube positions are randomly assigned during reset. 

---

## Cube Set

The system defines a pool of **nine cube variants**:

```
3 colors × 3 sizes
```

At the beginning of each episode:

1. Three cubes are sampled
2. They are placed on three slots
3. A command `(from_slot, to_slot)` is generated

Cube variants are defined as:

```
cube_light_s
cube_light_m
cube_light_l
cube_flat_s
cube_flat_m
cube_flat_l
cube_dark_s
cube_dark_m
cube_dark_l
```



---

# Robot

The environment uses:

```
UR10e robot arm
Robotiq mechanical gripper
```

The robot is configured using the official Isaac Lab articulation configuration. 

Control is performed through:

* Differential inverse kinematics for the arm
* Joint position control for the gripper

Actions include:

```
Arm Cartesian motion (IK)
Gripper opening / closing
NEXT signal action
```

The **NEXT action** allows the policy to signal that the manipulation command has been completed.

# Challenges Encountered

Several technical issues made the environment harder to stabilize than expected.

---

## Reward avoidance behaviour

If the robot pushes a cube, the cube may fall off the table.

When this happens, the agent loses the potential reward it could have earned by maintaining a safe configuration.

Because of this, the agent sometimes learns a conservative strategy:

```
stay above the cube without approaching
```

This behaviour is a common reinforcement learning phenomenon where the agent prefers avoiding risk instead of interacting with the object.

---

## Variable cube sizes

Cubes are generated with different scales.

This introduces inconsistencies between episodes:

Example:

```
episode A: cube height allows approach distance X
episode B: cube slightly larger → same distance pushes cube
```

This means the optimal policy can vary across episodes.

---

## Reach reward plateaus

Early reward formulations saturated far from the target, allowing the agent to reach local optima where it stopped improving.

Introducing **progress-based rewards** helped mitigate this issue.

---

## Simulation debugging complexity

Robotic RL environments are extremely sensitive to:

* frame definitions
* reward scaling
* physics parameters
* observation design

Several debugging sessions were required to identify issues such as incorrect TCP reference frames or unreachable reward conditions.

# Current Status

The project is **indefinitely paused**.

Although the environment works and reach behaviour was partially learned, the overall system became significantly more complex than initially intended.

Rather than continuing to iterate on this environment, I decided to return to **simpler projects** that better match my current experience level.
